# Memory System

**AI Media Intelligence OS — Memory System Philosophy and Implementation**

---

## Philosophy: Memory Is the Moat, Not the Model

The most common mistake in AI-powered content systems is treating the AI model as the competitive advantage. Models are commodities — they get better, cheaper, and more accessible every month. What creates lasting value is not which model you use, but what you remember.

Memory is the moat. The model is just the engine.

Consider two content systems using the same AI model:
- **System A** has no memory. Every content decision starts from scratch. It makes the same mistakes, repeats the same patterns, and never learns from outcomes.
- **System B** has deep memory. It knows which hooks generated the most engagement. It knows which topics resonate with the audience. It knows the optimal publishing window for each platform. It gets smarter with every article.

After 100 articles, System B will consistently outperform System A by a significant margin — even though they use the exact same model. That gap is the moat.

## Memory Architecture

The memory system has four key components:

1. **Storage**: Categorized memories with unique keys and performance scores
2. **Reinforcement Learning**: Score updates based on real analytics outcomes
3. **Pattern Detection**: Automatic discovery of high and low performing clusters
4. **Decay**: Gradual score reduction to prevent stale data from dominating

### Storage Model

Memories are stored in the `MemoryEntry` table with the following structure:

- **workspaceId**: Scope isolation — each workspace has its own memory universe
- **category**: What dimension this memory belongs to (hook, topic, tone, timing, cta, format, platform, monetization, audience, style)
- **key**: Unique identifier within the category (e.g., "controversial_question" for hook category)
- **value**: The actual memory content (e.g., "Opening with a controversial question increases CTR by 23%")
- **score**: Performance/confidence score (0-100)
- **source**: Where this memory came from (analytics, manual, ai, experiment)
- **contextJson**: Additional structured context
- **isActive**: Soft deactivation flag — memories are never deleted

The unique constraint `(workspaceId, category, key)` ensures there is exactly one memory per key per category per workspace. When a memory is upserted, it either creates a new entry or updates the existing one.

### Memory Categories

| Category | What It Stores | Why It Matters |
|----------|---------------|----------------|
| `hook` | Opening hooks and their performance | The first 3 seconds determine if someone reads. Knowing which hooks work eliminates guesswork. |
| `topic` | Topic resonance scores | Not all topics are equal. Some generate 10x the engagement. Memory tells you which. |
| `tone` | Voice/style effectiveness | Your audience responds differently to different tones. Memory tracks which tone works when. |
| `timing` | Optimal publishing windows | Publishing at the right time can double engagement. Memory learns your audience's schedule. |
| `cta` | Call-to-action effectiveness | The difference between "subscribe" and "join 10,000 others" is data, not intuition. |
| `format` | Content format performance | Listicles vs. essays vs. tutorials — memory knows what format your audience prefers. |
| `platform` | Platform-specific behavior | What works on WordPress may fail on Medium. Memory keeps platform-specific insights. |
| `monetization` | Revenue patterns | Which content generates revenue and how. Memory tracks the money trail. |
| `audience` | Audience segment insights | Different segments respond to different content. Memory knows the segments. |
| `style` | Writing style preferences | Sentence length, paragraph structure, vocabulary level — memory captures style patterns. |

### Memory Sources

| Source | Description | Trust Level |
|--------|-------------|-------------|
| `analytics` | Derived from real performance data | Highest — based on actual outcomes |
| `manual` | Manually entered by the operator | High — human judgment |
| `ai` | Generated by AI analysis | Medium — requires validation |
| `experiment` | From controlled A/B tests | High — controlled conditions (future) |

## Reinforcement Learning

The memory system uses a simple but effective reinforcement learning algorithm to update scores based on real outcomes.

### The Algorithm

When an analytics event arrives (e.g., "article X got 500 views on WordPress"), the `recordOutcome` function:

1. **Records the analytics event** in `AnalyticsEvent` (immutable)
2. **Finds all memory entries** related to the content's tags
3. **Updates scores** using the formula:
   ```
   newScore = oldScore + learningRate × (outcome - oldScore)
   ```
   Where `learningRate = 0.1` — slow and steady
4. **Clamps the score** to [0, 100]
5. **Stores a platform-specific performance memory** for future reference

### Why Learning Rate 0.1?

A learning rate of 0.1 means it takes approximately 10 positive outcomes for a memory's score to converge to the true value. This prevents:
- **Overreaction**: A single viral post doesn't make a pattern
- **Underreaction**: Consistent performance is recognized after enough data points
- **Noise**: Random fluctuations are smoothed out

### Example

A "controversial_question" hook memory has a score of 65. An article using this hook receives 85 engagement score:

```
newScore = 65 + 0.1 × (85 - 65) = 65 + 2 = 67
```

Next time, another article with this hook scores 90:

```
newScore = 67 + 0.1 × (90 - 67) = 67 + 2.3 = 69.3
```

After several positive outcomes, the score converges toward the true performance level. Negative outcomes pull the score down equally:

```
newScore = 69.3 + 0.1 × (40 - 69.3) = 69.3 - 2.93 = 66.37
```

## Pattern Detection

The `detectPatterns` function analyzes all active memories in a workspace to identify actionable patterns. It operates in two passes:

### Pass 1: Cluster Detection

For each memory category:
1. Find memories with score ≥ 70 (high performers)
2. If ≥ 2 high performers exist in a category, flag as a "high-performing cluster"
3. Find memories with score < 30 (low performers)
4. If ≥ 2 low performers exist in a category, flag as a "low-performing cluster"

### Pass 2: Timing Analysis

For the `timing` category:
1. Find the memory with the highest score
2. Flag as the "optimal publishing window"
3. Include confidence score based on the memory's score

### Output Format

Each detected pattern includes:
- **category**: The memory category
- **pattern**: Human-readable pattern description
- **confidence**: 0-1 confidence level
- **evidence**: Supporting evidence from memory entries
- **recommendation**: Actionable recommendation

### Example Output

```json
[

    "pattern": "High-performing hook cluster detected",
    "confidence": 0.87,
    "evidence": [

      "contrarian_take: 79.0"
    ],
    "recommendation": "Double down on hook patterns: controversial_question, data_surprise, contrarian_take"
```

## Memory Decay

Memory scores naturally decay over time via the `decayMemories` function. This prevents stale patterns from dominating decisions.

### Decay Mechanism

```typescript
newScore = oldScore × decayRate  // default decayRate = 0.95
```

This means each decay cycle reduces scores by 5%. After 14 cycles:
- Score 90 → 90 × 0.95^14 ≈ 43
- Score 50 → 50 × 0.95^14 ≈ 24
- Score 20 → 20 × 0.95^14 ≈ 10

### Deactivation

Memories that decay below a score of 1 are deactivated (`isActive = false`). They are not deleted — they remain in the database for historical analysis and potential reactivation.

### When to Decay

Decay should be run periodically (e.g., weekly or monthly). The frequency depends on content volume:
- **Low volume** (1-3 articles/week): Monthly decay
- **Medium volume** (4-10 articles/week): Bi-weekly decay
- **High volume** (10+ articles/week): Weekly decay

Decay is not automatic in V1 — it must be triggered via the memory API or scheduler.

## Platform Behavior Intelligence

The `getPlatformBehavior` function aggregates memory data for a specific platform, providing actionable insights:

### Output

```typescript
interface PlatformBehavior {
  platform: string;
  bestPerformingHooks: Array<{ key: string; value: string; score: number }>;
  bestPerformingTopics: Array<{ key: string; value: string; score: number }>;
  optimalTiming: Array<{ key: string; value: string; score: number }>;
  averageScore: number;
}
```

This function searches across all memory categories for entries related to the platform and aggregates them into a structured intelligence report. It enables the system to make platform-specific decisions rather than applying one-size-fits-all strategies.

## Memory in the Content Pipeline

Memory is injected into the content pipeline at two critical points:

### 1. Before Draft Generation

When generating a new draft, the system retrieves:
- Top hooks from memory (to inspire the opening)
- Top topics from memory (to validate the idea)
- Platform behavior (to optimize for the target platform)

This context is included in the AI prompt, ensuring new content benefits from accumulated knowledge.

### 2. After Analytics Arrive

When analytics data comes in, the system:
1. Records the outcome
2. Updates memory scores via reinforcement learning
3. Runs pattern detection to discover new insights
4. Stores platform-specific performance memories

This closes the feedback loop: content decisions are informed by memory, and outcomes update memory.

## Memory vs. Logging

It is important to distinguish memory from logging:

| Aspect | Memory | Logging (SystemLog) |
|--------|--------|-------------------|
| **Purpose** | Inform future decisions | Debug and audit |
| **Content** | What worked, what failed | What happened, when, and why |
| **Retention** | Indefinite (with decay) | Indefinite (with purge) |
| **Querying** | By category, key, score | By service, level, action |
| **Action** | Drives content decisions | Drives debugging |
| **Structure** | Typed categories with scores | Unstructured metadata |

Both are essential, but they serve different purposes. Memory makes the system smarter. Logging makes the system debuggable.
