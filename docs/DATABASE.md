# Database Schema Documentation

**AI Media Intelligence OS — Canonical Database Schema**

---

## Overview

The database schema is the canonical definition of all entities in the AI Media Intelligence OS. It uses Prisma ORM with SQLite as the V1 database engine, with a clear migration path to PostgreSQL for production scaling.

Every table is designed around the core principle of **workspace isolation**: all data belongs to a workspace, and workspace deletion cascades to all related records. This ensures clean multi-tenancy and predictable data ownership.

The schema follows the "one master, many variants" content model. There is exactly one `ContentItem` per idea, and zero or more `ContentVariant` records derived from it. This prevents content fragmentation and ensures there is always a single source of truth.

#### Status Lifecycle

```
idea → draft → editing → seo_review → ready → scheduled → published → archived
                                                                ↓
                                                             failed
```

- **idea**: Initial concept, no content yet
- **draft**: Master markdown generated by AI
- **editing**: Humanic rewrite applied, being refined
- **seo_review**: SEO pack generated, awaiting review
- **ready**: Scored and approved for publishing
- **scheduled**: Queued for publishing at a specific time
- **published**: Successfully published to platform
- **archived**: No longer active, retained for memory
- **failed**: Publishing failed after max retries

## Schema Design Principles

1. **Workspace isolation first**: Every query includes `workspaceId` — no cross-workspace data leaks
2. **Cascade deletes**: Workspace deletion removes all related data cleanly
3. **Append-only analytics**: Analytics events and system logs are never mutated
4. **Soft deactivation**: Memory entries use `isActive` rather than deletion, preserving history
5. **JSON flexibility**: Complex configuration uses `Json` fields with documented schemas
6. **Index-heavy**: Every common query pattern has a dedicated index
7. **CUID identifiers**: Collision-resistant, URL-safe, sorted by creation time
8. **Timestamps on everything**: `createdAt` and `updatedAt` on all mutable entities
